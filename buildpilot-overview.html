What Is BuildPilot

Product Vision

One-liner: BuildPilot is an AI-guided product development OS that helps independent builders turn an idea into real, maintainable software — by guiding the process, preserving decisions, generating living documentation, and coordinating work across external AI tools.

The core insight: AI coding tools (Cursor, Claude Code, v0, Lovable) are getting very good at generating code, but they are terrible at building products. They have no memory, no process, no structure — every session starts cold. BuildPilot is the missing layer.

Challenges

Retrospective
🛤️
Windows Path Spaces → shadcn CLI Failure
The documents folder path (C:\Users\User\OneDrive - Center For Education Technology\Documents\MindCET\Tandem - Claude Code) contains spaces. The shadcn/ui CLI fails to resolve modules on paths with spaces on Windows.
✅ Resolved: Moved all code to C:\dev\buildpilot. Planning docs stay in original location.
🎨
Tailwind v4 vs v3 API Differences
Next.js 16 installs Tailwind CSS v4 by default. Tailwind v4 has a completely different config API — no tailwind.config.ts, no @tailwind base/components/utilities imports. The plan.md was written assuming v3. Many snippets in the plan reference v3 patterns.
⚠️ Pending: All Phase 1+ CSS patterns need to be written for v4. Use @import "tailwindcss" in globals.css, not the v3 directives.
⚛️
Next.js 16 / React 19 vs Planned 14/18
The original plan specified Next.js 14 and React 18. The installed version is Next.js 16.2.6 with React 19. React 19 has breaking changes in form handling and some server action patterns differ.
✅ Low risk: Core App Router patterns are compatible. Supabase SSR patterns still work. React 19 improvements (Actions, useFormStatus) are actually beneficial.
🌐
Hebrew RTL Layout Complexity
The UI is bilingual — Hebrew labels with English technical terms. Hebrew is RTL but the technical content (code blocks, file paths, API routes) must remain LTR. shadcn/ui components are not RTL-optimized by default. The sidebar layout and flex directions need careful handling.
⚠️ Pending: Add dir="rtl" to root layout for Hebrew sections, use dir="ltr" override on code/technical elements. Test nav sidebar rendering.
🤖
AI JSON Output Reliability
Claude must return structured JSON matching Zod schemas. Even with explicit prompting, AI models occasionally return JSON wrapped in markdown code fences, add extra explanation text, or deviate from the schema. Failures crash the service layer.
⚠️ Pending: Add JSON extraction (strip code fences), retry logic (max 2 retries), and graceful fallback messages. Consider using anthropic.messages.create with tool_use for guaranteed JSON.
🔑
Supabase Environment Variables — Local vs Production
The Supabase project hasn't been created yet (Phase 1 task 1-1). The .env.local has placeholder values. Until a real Supabase project exists, the app will fail to start in any mode that touches auth.
⚠️ Blocker for Phase 1: Create Supabase project first, then fill .env.local and Vercel env vars.
💰
AI API Cost at Scale
Every "Generate Brief", "Generate PRD", "Generate Architecture" call costs tokens. A full project run (brief → PRD → arch → tasks → missions → return analysis) could cost $0.50–$3.00 per project in API fees. With many users, costs compound.
⚠️ Planned: Add prompt caching for repeated context (decisions, project data). Rate limit AI endpoints per user. Show token usage in AI logs table for transparency.

AI Service Layer

The core pattern: every AI feature goes through callAI() — a generic wrapper that handles prompting, JSON parsing, and Zod validation. Review before Phase 2.

lib/ai/service.tsTypeScript
import { anthropic, MODEL } from './client'
import { ProductBriefSchema, ClarificationQuestionsSchema } from './schemas'
import { PRODUCT_BRIEF_SYSTEM, CLARIFICATION_SYSTEM } from './prompts'
import type { ZodSchema } from 'zod'

// Generic AI caller — all AI features go through here
async function callAI<T>(
  schema: ZodSchema<T>,
  systemPrompt: string,
  userMessage: string,
): Promise<T> {
  const response = await anthropic.messages.create({
    model: MODEL,
    max_tokens: 4096,
    system: systemPrompt,
    messages: [{ role: 'user', content: userMessage }],
  })

  const raw = response.content[0].type === 'text' ? response.content[0].text : ''

  // Strip markdown code fences if Claude wraps in ```json ... ```
  const cleaned = raw.replace(/^```(?:json)?\n?/m, '').replace(/\n?```$/m, '').trim()

  const parsed = JSON.parse(cleaned)
  return schema.parse(parsed) // Zod throws if schema mismatch
}

export async function generateClarificationQuestions(
  idea: string,
  context: Record<string, string>
) {
  return callAI(
    ClarificationQuestionsSchema,
    CLARIFICATION_SYSTEM,
    `Product idea: ${idea}\n\nAdditional context: ${JSON.stringify(context, null, 2)}`
  )
}

export async function generateProductBrief(
  idea: string,
  answers: Record<string, string>
) {
  return callAI(
    ProductBriefSchema,
    PRODUCT_BRIEF_SYSTEM,
    `Product idea: ${idea}\n\nClarification answers: ${JSON.stringify(answers, null, 2)}`
  )
}

Supabase Server Client (SSR-safe)

This is the pattern for all server-side Supabase access. Uses cookies for session management — critical for Next.js App Router. Never use the browser client in Server Components.

lib/supabase/server.tsTypeScript
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'
import type { Database } from '@/types/database'

export async function createClient() {
  const cookieStore = await cookies()

  return createServerClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return cookieStore.getAll()
        },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options))
          } catch {
            // setAll called from a Server Component — cookies() is read-only
            // Safe to ignore — middleware handles session refresh
          }
        },
      },
    }
  )
}

Return Analysis API Route — the Heart of SPDD

When a user pastes their return brief from Cursor/v0/Claude Code, this route: parses it, extracts decisions + risks, updates the database, and advances the project stage. Most critical route in the product.

app/api/ai/analyze-return/route.tsTypeScript
import { NextRequest, NextResponse } from 'next/server'
import { createClient } from '@/lib/supabase/server'
import { analyzeReturnBrief } from '@/lib/ai/service'

export async function POST(req: NextRequest) {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })

  const { projectId, missionId, rawSummary } = await req.json()

  // Fetch project context for the AI
  const [projectRes, decisionsRes, tasksRes] = await Promise.all([
    supabase.from('projects').select('*').eq('id', projectId).single(),
    supabase.from('decisions').select('title, decision').eq('project_id', projectId).eq('status', 'active').limit(10),
    supabase.from('tasks').select('title, acceptance_criteria').eq('id', /* current task */ missionId).single(),
  ])

  const analysis = await analyzeReturnBrief(rawSummary, {
    project: projectRes.data,
    decisions: decisionsRes.data,
    task: tasksRes.data,
  })

  // Auto-save decisions
  if (analysis.new_decisions.length > 0) {
    await supabase.from('decisions').insert(
      analysis.new_decisions.map(d => ({ ...d, project_id: projectId }))
    )
  }

  // Auto-save risks
  if (analysis.risks.length > 0) {
    await supabase.from('risks').insert(
      analysis.risks.map(r => ({ ...r, project_id: projectId, source: 'return_brief' }))
    )
  }

  // Save tool session record
  await supabase.from('tool_sessions').insert({
    project_id: projectId,
    mission_id: missionId,
    raw_summary: rawSummary,
    parsed_summary: analysis,
    status: 'analyzed',
  })

  // Update mission status
  await supabase.from('missions')
    .update({ status: 'returned', returned_at: new Date().toISOString() })
    .eq('id', missionId)

  return NextResponse.json({ analysis })
}

Companion Context Builder

The AI Companion always knows your project state. This function builds the context string injected into every companion message. Keep this fast — it runs on every chat message.

lib/ai/companion/context.tsTypeScript
import type { SupabaseClient } from '@supabase/supabase-js'
import type { Database } from '@/types/database'

export async function buildProjectContext(
  projectId: string,
  supabase: SupabaseClient<Database>
): Promise<string> {
  // Parallel fetch for performance
  const [project, artifacts, decisions, tasks, openRisks] = await Promise.all([
    supabase.from('projects').select('name, current_stage, description').eq('id', projectId).single(),
    supabase.from('artifacts').select('type, title, status').eq('project_id', projectId),
    supabase.from('decisions').select('title, decision, category').eq('project_id', projectId).eq('status', 'active').limit(8),
    supabase.from('tasks').select('title, status, recommended_tool, phase').eq('project_id', projectId).order('created_at').limit(15),
    supabase.from('risks').select('title, severity').eq('project_id', projectId).eq('status', 'open').eq('severity', 'high').limit(3),
  ])

  const nextTask = tasks.data?.find(t => t.status === 'not_started' || t.status === 'ready')

  return `
## Project: ${project.data?.name}
Stage: ${project.data?.current_stage}
Description: ${project.data?.description}

## Artifacts (${artifacts.data?.length} total)
${artifacts.data?.map(a => `- ${a.type} → ${a.title} [${a.status}]`).join('\n') ?? 'None yet'}

## Key Decisions
${decisions.data?.map(d => `- [${d.category}] ${d.title}: ${d.decision}`).join('\n') ?? 'No decisions recorded'}

## Tasks
${tasks.data?.map(t => `- ${t.title} [${t.status}] → tool: ${t.recommended_tool}`).join('\n') ?? 'No tasks yet'}

## Next Task
${nextTask ? `${nextTask.title} (use ${nextTask.recommended_tool})` : 'All tasks complete or none generated yet'}

## Open High-Risk Issues
${openRisks.data?.map(r => `- ${r.title} [${r.severity}]`).join('\n') ?? 'None'}
  `.trim()
}
